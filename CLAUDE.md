# CloneWeb — Codebase Guide for AI Assistants

## Project Overview

CloneWeb is an ASP.NET Core 6.0 MVC blogging/content management platform. It supports user authentication, blog post creation with rich-text editing, commenting, category and tag management, and real-time visitor tracking via SignalR.

**Database name:** `BlogWeb` (SQL Server)  
**Launch URL (dev):** `https://localhost:44370/` (IIS Express) or `https://localhost:5001/` (Kestrel)

---

## Solution Structure

```
CloneWeb.sln
├── CloneWeb/              Main ASP.NET Core MVC web application
├── EntityDataModel/       EF Core data layer — DbContext + auto-generated entity models
├── ViewModel/             DTOs passed between controllers and views
├── Core/                  Shared base controller (BaseController.cs)
└── Extension/             Utility extension library (currently empty)
```

Each subdirectory contains a `.csproj` file. The web project depends on all others.

---

## Development Commands

```bash
# Restore dependencies
dotnet restore CloneWeb.sln

# Build the solution
dotnet build CloneWeb.sln

# Run the web app (Kestrel)
dotnet run --project CloneWeb/CloneWeb.csproj

# Build release
dotnet build CloneWeb.sln -c Release
```

There is no test project — `dotnet test` will find nothing to run.

Client-side libraries (jQuery, Bootstrap, SignalR) are managed by **LibMan** (`CloneWeb/libman.json`), not npm. The `package.json` at the root only has a single CKEditor dependency and no build scripts.

## Architecture

The application follows a standard **ASP.NET Core MVC** pattern with clear project boundaries:

```
HTTP Request
    → Controller (CloneWeb/Controllers/)
        → EF Core DbContext (EntityDataModel/Data/EntityDataContext.cs)
            → SQL Server (database: BlogWeb)
        → ViewModel (ViewModel/) passed to View
    → Razor View (CloneWeb/Views/)
        → Response
```

### Projects in detail

**CloneWeb/** — Web application entry point
- `Program.cs` — creates and runs the host
- `Startup.cs` — registers services (EF Core, cookie auth, SignalR, MVC) and builds the middleware pipeline
- `Controllers/` — 5 controllers: Home, Post, Authentication, About, Error
- `Views/` — Razor templates (`.cshtml`), shared layout, and view components
- `Middleware/` — `VisitorCounterMiddleware.cs` (cookie-based visitor tracking; currently disabled in Startup)
- `wwwroot/` — static assets (CSS, JS, uploaded files, third-party plugins)

**EntityDataModel/** — Data layer
- `Data/EntityDataContext.cs` — the single DbContext; auto-generated by EF Core Power Tools
- `Models/` — 7 entity classes: `Post`, `User`, `Category`, `Tag`, `Comments`, `PostComment`, `PostTag`

**ViewModel/** — Data transfer objects
- `Create/PostViewModel.cs` — extends `Post` entity, adds `TagId` list for multi-tag selection
- `ResultViewModel/PostResultViewModel.cs` — enriched post data for display (author name, tag list, comment count)
- `ResultViewModel/CommentResultViewModel.cs` — comment display model
- `PagingViewModel.cs`, `CookieAuthenticationModel.cs`, `Reponse.cs` — supporting models

**Core/**
- `BaseController.cs` — base class for controllers (15 lines; extend here for shared controller logic)

## Database & Data Layer

**ORM:** Entity Framework Core 6.0.10  
**Connection string key:** `ConnectionStrings:DefaultConnection` in `appsettings.json`

```
Development: Data Source=RIN;Initial Catalog=BlogWeb;Integrated Security=True
Production:  Azure SQL Server (commented out in appsettings.json)
```

### Entity model summary

| Entity | Table | Key | Notes |
|--------|-------|-----|-------|
| `Post` | Post | PostId (Guid) | Title, Content (PostInfomation), URL slug, CategoryId, CreateBy, CreateTime |
| `User` | User | UserId (Guid) | UserName, Email, Password (MD5), Role, ImageUrl |
| `Category` | Category | CategoryId (Guid) | Title, ParentId (self-referencing) |
| `Tag` | Tag | TagId (Guid) | Title |
| `Comments` | Comments | CommentId (Guid) | CommentMessage, CreateBy → User FK |
| `PostComment` | Post_Comment | (PostId, CommentId) composite | Join table for Post ↔ Comments |
| `PostTag` | Post_Tag | (PostId, TagId) composite | Join table for Post ↔ Tags |

The `EntityDataContext` is **auto-generated** — do not hand-edit it. Regenerate using EF Core Power Tools if the schema changes.

### Typical query pattern

Controllers inject `EntityDataContext` directly and use LINQ:

```csharp
var post = await (from db in _context.Post.Include(x => x.PostTag)
                  join cate in _context.Category on db.CategoryId equals cate.CategoryId
                  where db.PostId == PostId
                  select new PostResultViewModel { ... }).FirstOrDefaultAsync();
```

Always use `async/await` with EF Core (`ToListAsync`, `FirstOrDefaultAsync`, `SaveChangesAsync`).

## Authentication & Authorization

**Scheme:** Cookie-based (`CookieAuthenticationDefaults.AuthenticationScheme`)  
**Session duration:** 60 minutes, sliding expiration enabled  
**Login path:** `/Authentication/Login`  
**Logout path:** `/Authentication/Logout`  
**Access denied path:** `/Forbidden/`

JWT Bearer is fully configured in `Startup.cs` and `appsettings.json` but commented out — do not remove.

### User identity in controllers

After login, user identity is stored as claims. Access them like this:

```csharp
var claims = User?.Identities.First().Claims.ToList();
var userId = Guid.Parse(claims.Where(x => x.Type == "UserId").First().Value);
```

Claims set at login: `UserId`, `UserName`, `Role`.

Use `[Authorize]` on actions or controllers that require authentication.

## Routing

Default convention-based route:
```
{controller=Home}/{action=Index}/{id?}
```

Custom route for viewing a post (defined on `PostController.ViewPost`):
```
{Year}/{Month}/{Date}/{Url}/{PostId}
```
Example: `/2024/01/15/my-first-post/3fa85f64-...`

The `Url` slug is generated by `PostController.ToUrlSlug(string value)` — lowercases the title, strips accents, replaces spaces with `-`, and removes invalid characters.

## Views & Frontend

**View engine:** Razor (`.cshtml`)  
**Master layout:** `CloneWeb/Views/Shared/_Layout.cshtml`  
**Frontend libraries** (managed by LibMan, not npm):
- Bootstrap 5
- jQuery 3
- SignalR client

**Third-party plugins** (in `wwwroot/Plugin/`):
- CKEditor 5 Classic — rich text editor for post content
- CKFinder 3.5.2 — file/image manager integrated with CKEditor

**View Components** (in `CloneWeb/Views/ViewComponents/`):
- `ListCommentsViewComponent` — renders comments for a post
- `ListRecentPostViewComponent` — sidebar: 5 most recent posts
- `ListCategoryViewComponent` — sidebar: category list
- `ListRecentCommentViewComponent` — sidebar: recent comments
- `TotalVisitViewComponent` — visit statistics
- `OnlineCountHub` — SignalR hub (endpoint: `/onlinecount`) for real-time online visitor count

### File uploads

Images uploaded via CKEditor go to `wwwroot/Upload/Post/`. The upload handler is in `HomeController.UploadImage()`, which returns `{ url: "<path>" }` as JSON.

## Code Conventions

- **Dependency injection:** Constructor injection is used everywhere. Add new services in `Startup.ConfigureServices`.
- **Async/await:** All database operations and I/O should be async.
- **ViewBag:** Used to pass auxiliary data (dropdowns, config values) from controllers to views.
- **Model validation:** Check `ModelState.IsValid` in POST handlers before processing.
- **Anti-forgery:** `[ValidateAntiForgeryToken]` is required on POST actions that mutate state.
- **GUID primary keys:** All entities use `Guid` PKs generated in application code (`Guid.NewGuid()`), not database-generated.
- **No repository layer:** Controllers access `EntityDataContext` directly.

## Configuration

All runtime config lives in `appsettings.json`. Key entries:

```json
{
  "ConnectionStrings": {
    "DefaultConnection": "<SQL Server connection string>"
  },
  "DomainUrl": "https://localhost:44370/",
  "Jwt": {
    "Key": "...",
    "Issuer": "...",
    "Audience": "..."
  }
}
```

Read values in controllers via `IConfiguration`:
```csharp
var domain = _configuration["DomainUrl"];
```
